# 理解项目

系统化地引导用户理解一个新项目或不熟悉的代码库，从全局概览到动手实践。

## 核心原则

> 建立可工作的心智模型，而非追求完全理解。理解 60% 就可以开始动手——剩下的在实践中补齐。

## 进度持久化

每个阶段结束后，将阶段产出写入 `docs/onboarding.md`。这是硬性要求，不是可选步骤。

### 恢复流程

开始时先检查 `docs/onboarding.md` 是否存在且未完成：
1. **不存在** → 正常从阶段一开始
2. **存在且未完成** → 读取文件，展示进度表，告诉用户上次停在哪。问用户："要继续阶段 X，还是从头开始？"
   - 用户选择继续 → 给一个简短摘要回顾前面阶段的产出，然后从断点继续
   - 用户选择从头开始 → 清空文件，重新开始
3. **存在且已完成** → 告知用户这个项目已经完成过上手流程，问是否要重新开始

### 完成后

五阶段全部完成后，在进度表下方加一行：`> 上手流程已完成。此文件作为学习记录保留，不再维护。`

## 五个阶段

默认按顺序执行，但根据用户情况和项目状态可以跳过或调整深度（见"自适应规则"）。每个阶段的流程：探索代码库 → 整理发现 → 用通俗语言呈现给用户 → 告诉用户自检方式。每个阶段结束后将产出写入 `docs/onboarding.md` 并更新进度表，然后询问用户是否继续下一阶段。

### 阶段一：全局认知（目标 5-10 分钟）

**目标：** 用户能一句话描述这个项目，并掌握项目的核心术语。

先做三件事再深入：

1. **先读文档再读代码** — 检查项目是否有 `README.md`、`CONTRIBUTING.md`、`CLAUDE.md`、`ARCHITECTURE.md`、`docs/` 目录或文件。文档是最快理解项目的入口。特别留意是否有架构决策记录（`docs/adr/`、`decisions/`），它们记录了"为什么这么设计"，比看代码猜意图高效得多。
2. **了解项目活跃度和核心模块** — 根据是否为 git 仓库选择方式：
   - **是 git 仓库：** 扫一眼 git 历史，看三样：
     - 提交频率判断活跃度（`git log --oneline --since="3 months ago" | wc -l`）
     - 高频改动文件识别核心模块（`git log --format="" --name-only | sort | uniq -c | sort -rn | head -20`）
     - 贡献者集中度判断是否有人特别懂这个项目（`git shortlog -sn`）
   - **非 git 仓库：** 用替代方式了解：
     - 检查是否有 `CHANGELOG.md`、`HISTORY.md`、`RELEASES.md` 等变更日志
     - 按修改时间排序文件（`find . -type f -not -path '*/node_modules/*' -not -path '*/.git/*' | xargs ls -lt 2>/dev/null | head -30`）识别近期活跃的文件
     - 从文档中提取项目阶段和活跃度信息
3. **判断项目规模** — 根据文件数量和目录深度调整后续阶段的深度：
   - 小型（<100 文件）：每个阶段可以深入看更多文件
   - 中型（100-1000 文件）：聚焦核心模块，跳过辅助模块
   - 大型（>1000 文件）或 monorepo：先识别 workspace 中的核心 package/service，只深入一个，其余只看公开接口
   - **Monorepo 特别注意** — 检查是否有 `packages/`、`apps/`、`services/` 目录或 workspace 配置（`pnpm-workspace.yaml`、`lerna.json`、`go.work`）。先引导用户识别哪个 package 是入口/核心，避免在几十个包里迷失。

然后探索并回答：
- 这个项目解决什么问题？给谁用的？
- 技术栈是什么？（检查 `package.json`、`go.mod`、`Cargo.toml`、`pom.xml`、`requirements.txt` 等）
- 项目处于什么阶段？（早期/成熟/维护期——结合 git log 提交频率判断）
- 顶层目录结构——每个目录一句话说明职责
- **项目术语表** — 每个项目有自己的黑话（业务术语、缩写、内部概念）。从 README、注释、变量名、API schema、数据库表名、配置项 key 中提取出来，列一份简表。不懂术语就看不懂代码和注释。

**重点文件：** `README.md`、`CONTRIBUTING.md`、`docs/adr/`、`decisions/`、依赖文件、顶层目录

**输出格式：**
```
项目：[一句话描述]
技术栈：[列表]
阶段：[判断]
规模：[小型/中型/大型/Monorepo]
目录地图：[结构化列表]
术语表：[关键术语 → 含义]
```

**自检：** 用户能否回答"这个项目是 [什么]，用 [哪些技术]，解决 [谁] 的 [什么问题]"？项目里的核心术语都理解了吗？

询问："要继续阶段二（把项目跑起来）吗？"

### 阶段二：让项目跑起来（可选，目标 10-30 分钟）

**目标：** 项目在本地能运行，测试能通过。

> 本阶段可选。大型/遗留项目本地运行可能耗时很长（配数据库、装依赖等），理解架构不依赖本地运行。如果遇到阻碍，跳过直接进阶段三，后续再回来。

探索并引导：
- 环境要求（语言版本、数据库等）
- 安装依赖、配置环境
- 启动项目
- 跑一遍测试——报告状态

**重点文件：** `Makefile`、`Dockerfile`、`docker-compose.yml`、CI 配置（`.github/workflows/`、`.gitlab-ci.yml`）、`.env.example`

**如果跑不起来：** 不是所有项目都能轻松本地运行（环境复杂、硬件依赖、只读分析等）。如果跑不起来，告诉用户跳过不影响后续理解，直接进阶段三。必要时使用 `diagnose` skill，如果该 skill 不可用，则引导用户逐步排查环境问题。

**自检：** 项目能跑起来，测试能通过，用户能访问到它（打开页面/调通接口）。如果跳过了，确保用户知道怎么在需要时回来。

> **回溯提示：** 完成阶段三后对项目理解更深，此时可以回来再试阶段二，很多之前不理解的配置项现在会有意义。

询问："要继续阶段三（核心数据流）吗？"

### 阶段三：核心数据流与架构（目标 15-30 分钟）

**目标：** 用户能在纸上画出项目的核心流转路径，理解模块间的边界和通信方式，识别关键设计模式。

**最关键的一步——真正的理解在这里发生。**

先识别项目类型，选择正确的切入点：

| 类型 | 切入点 | 追踪路径 |
|---|---|---|
| 后端 API | 路由定义 | 路由 → 中间件 → Controller → Service → 数据库 |
| 前端 SPA | 入口文件（`main.ts`、`App.vue`） | 入口 → 路由 → 页面 → 状态管理 → API 调用 |
| CLI / 工具 | 命令入口（`cmd/`、`cli.py`） | 命令 → 参数解析 → 核心逻辑 → 输出 |
| 库 / SDK | 公开 API / 导出 | 公开接口 → 内部实现 |
| AI / ML 项目 | 训练/推理入口（`train.py`、`infer.py`、`main.ipynb`） | 数据加载 → 预处理 → 模型定义 → 训练/推理 → 输出 |
| 算法项目 | 主算法入口（`solve()`、`main()`） | 输入定义 → 核心算法 → 数据结构 → 输出/验证 |
| 数据工程 | Pipeline 入口（`etl.py`、`dag.py`、`spark_job.py`） | 数据源 → 清洗/转换 → 存储 → 调度 |
| 移动端（React Native / Flutter） | 入口（`App.tsx`、`main.dart`） | 入口 → 导航 → 页面 → 状态管理 → 原生桥接 |
| Serverless / 微服务 | API Gateway / 函数入口（`handler.py`、`index.ts`） | 触发器 → 函数入口 → 业务逻辑 → 外部服务调用 |
| 插件系统 | 插件注册/加载机制（`plugin.py`、`extensions/`） | 插件接口定义 → 注册流程 → Hook/事件绑定 → 插件实现 |
| Monorepo | Workspace 根配置 + 核心 package | 根配置 → package 依赖图 → 核心 package 入口 → 各 package 公开接口 |
| 其他 / 不确定 | 从 README 或入口文件找线索 | 从调用方到被调用方，逐层追踪 |

然后探索：
1. 找到入口文件
2. **对大型项目，启动多个 Explore agent 并行分析不同模块**（例如后端和前端分别追踪），最后汇总结果。中小型项目则顺序分析即可。
3. 找到核心抽象——不限于数据模型，还可能是算法核心、模型定义、数据结构：
   - Web 项目：`models/`、`types/`、`schema/`、`entities/`
   - AI/ML 项目：模型定义（`model.py`、`network.py`）、训练配置、数据处理流程
   - 算法项目：核心数据结构、算法实现、复杂度相关的关键设计
   - 数据工程：数据 schema、转换规则、上下游依赖
3. **通过测试理解行为** — 找到核心模块的测试文件，测试是最好的"需求文档"，它告诉你代码应该怎么表现。理解测试用例覆盖的场景，再去看实现。如果项目没有测试（早期项目、POC、脚本等），改为通过类型签名、接口文档、或直接读实现来理解行为。
4. 挑一个最重要的功能，追踪完整调用链
5. 确定数据的生命周期：从哪来、怎么变、存到哪去
6. **追踪失败路径** — 不只看 happy path，还要看错误怎么产生、怎么传播、怎么被处理。理解失败路径才能写出健壮的代码。
7. **识别模块边界** — 模块之间是怎么通信的？（函数调用、事件、消息队列、HTTP、共享状态？）接口契约是什么？
8. **识别设计模式** — 项目用了哪些设计模式（Observer、Strategy、Factory、Middleware、Repository、Plugin 等）？列出识别到的模式、用在哪里、解决什么问题。设计模式是理解设计意图的快捷方式——识别出模式就理解了一半。

**输出（三件套）：**

1. **架构图** — 优先使用 `excalidraw-diagram` skill 画出模块关系和数据流。如果该 skill 不可用，使用 ASCII 文本图或 Mermaid 语法作为 fallback。画图是标准动作，不是可选项。
2. **关键文件清单** — 列出最值得精读的 5-10 个文件，每个文件一句话说明为什么重要。比漫无目的地看代码更高效。
3. **设计模式清单** — 列出识别到的模式、位置和意图。

**自检：** 用户能否在纸上画出项目的核心流转路径？能否说清模块之间的通信方式？能否指出项目用了哪些设计模式？

询问："要继续阶段四（约定与边界）吗？"

### 阶段四：约定与边界（目标 10-15 分钟）

**目标：** 用户理解项目的"潜规则"和边界条件——知道什么能改、什么不能碰、为什么这么设计。

探索并讲解（按优先级排列）：

**必看：**
- 关键第三方依赖及其作用（避免把依赖做的事当成项目代码来理解）
- 已知问题和技术债（检查 TODO 注释、README 中的已知问题、issue tracker）
- 认证和权限模型（安全相关，不了解就不能乱改）

**重要：**
- 代码风格和命名规范（lint 配置、既有模式）
- 错误处理方式（try/catch、中间件、result 类型？）
- 配置管理（`.env`、配置文件、feature flag）

**了解即可：**
- 日志和监控方式
- 并发/异步逻辑（这类 bug 难复现，必须提前理解设计意图）
- 部署流程和环境差异（本地跑得通 ≠ 生产跑得通）

**重点文件：** lint 配置（`.eslintrc`、`pyproject.toml`、`golangci.yml`）、middleware/、`.env.example`、日志配置、CI 配置

**自检：** 用户能否回答"如果我改了 [某个模块]，可能会影响什么？哪些地方是绝对不能动的？"

询问："要继续阶段五（动手实践）吗？"

### 阶段五：动手实践（目标 15-30 分钟）

**目标：** 通过完成一个小任务，把前四个阶段的"知道"变成真正的"会做"。

根据项目状态和用户水平，按以下优先级选择任务：

1. **项目有 "good first issue" 标签或已知小 bug（TODO 注释、issue tracker）？** → 优先修一个小 bug。最贴近真实工作，成就感最强。
2. **项目有测试但覆盖不全？** → 给未覆盖的核心函数写测试（使用 `tdd` skill）。写测试迫使理解函数的输入输出和边界条件。
3. **以上都不满足？** → 选一个核心功能，从入口追踪到实现，边追踪边注释自己的理解。不依赖外部资源，任何项目都能做。

备选任务（根据情况穿插）：
- **在核心流程加调试日志** — 观察数据如何流转变化
- **做一个小改动** — 加一个字段、调整排序、改一段文案

逐步帮助用户完成任务。任务完成后，帮用户总结：改了什么、为什么这么改、涉及了哪些模块。

## 自适应规则

- **用户说"我是初学者"：** 不用术语。多用类比。解释每个不明显的概念。在阶段三多花时间。
- **用户说"我需要快速上手干活"：** 压缩阶段一二。只聚焦用户工作相关的阶段三，然后直接进阶段五。
- **用户说"我在评估这个项目"：** 聚焦阶段一和三。跳过阶段五。额外评估代码质量、架构健康度、维护风险。
- **用户只想理解某个模块/文件：** 跳过阶段一二，对该模块做迷你版阶段三（找入口、找核心抽象、追踪调用链），然后直接进阶段五针对该模块做练习。
- **用户有具体问题：** 直接跳到相关阶段，不强制按顺序。

## 常见陷阱提醒

当发现用户陷入以下模式时，主动提醒：
- 试图看懂每一行代码 → "先看主干路径，分支逻辑以后再看"
- 从第一行代码开始顺序读 → "从入口开始，沿着调用链往下读"
- 只看代码不动手 → "试试加个日志跑一下，看一次数据流转比读 100 行代码更管用"
- 只问"是什么"不问"为什么" → "问问为什么这么设计，而不只是做了什么"
- 想完全理解再开始 → "理解 60% 就够了——剩下的在实践中学习"

## 输出语言

匹配用户的语言。用中文问就用中文答，用英文问就用英文答。
